# Bugfix Requirements Document

## Introduction

The module steering files (module-01 through module-12, plus module-completion.md and module-07-reference.md) contain inline 👉 closing questions with WAIT instructions that conflict with the `ask-bootcamper` hook's role as the single owner of closing questions. This is the same architectural conflict that was already fixed in `onboarding-flow.md` (see `.kiro/specs/missing-pointing-prefix/`). The `agent-instructions.md` rule states: "Never end your turn with a closing question — the `ask-bootcamper` hook owns all closing questions." However, module steering files instruct the agent to ask inline 👉 questions and then WAIT for a response, causing the agent to interpret "WAIT for response" as "ask the question myself and wait" rather than "present the information and let the hook ask." During Module 1, the bootcamper had to correct this behavior at least 6 times, and the agent repeated the mistake even after acknowledging it — because the steering file instructions directly contradicted the hook ownership rule.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN the agent executes a module step that ends with an inline 👉 closing question and WAIT instruction (e.g., module-01 step 1: "Would you like me to initialize a git repository?" WAIT for response) THEN the system has two competing sources of closing questions — the inline question from the steering file and the `ask-bootcamper` hook — creating a question-ownership conflict that the agent cannot reliably resolve

1.2 WHEN the agent encounters a WAIT instruction after an inline question in a module steering file THEN the agent interprets "WAIT for response" as "ask the question myself and wait" rather than "present the information and stop so the hook can ask," causing the agent to place a closing 👉 question at the end of its output in violation of the `agent-instructions.md` ownership rule

1.3 WHEN the agent's turn ends with an inline 👉 question from a module steering file AND the `ask-bootcamper` hook fires on `agentStop` THEN the hook may generate a second 👉 question because prompt-based suppression ("If your previous output already ends with a 👉 question, do nothing") is not reliably followed by LLMs, resulting in duplicate back-to-back questions

1.4 WHEN the agent paraphrases an inline 👉 question from a module steering file THEN the system may drop the 👉 prefix because the question is embedded in steering file prose rather than being generated by the hook's structured prompt

1.5 WHEN the bootcamper corrects the agent for placing a closing question and the agent acknowledges the mistake THEN the agent repeats the same mistake on the very next turn because the module steering file still instructs it to ask the question and WAIT — the steering file instruction overrides the agent's acknowledgment of the rule

### Expected Behavior (Correct)

2.1 WHEN the agent executes any module step THEN the module steering file SHALL NOT contain inline 👉 closing questions or WAIT-for-response instructions at the end of steps; the agent SHALL present the step's information and then stop, allowing the `ask-bootcamper` hook to be the sole source of the closing 👉 question

2.2 WHEN a module step requires user input before proceeding (e.g., choosing a database, selecting a loading strategy, confirming a plan) THEN the steering file SHALL describe the choices or information to present without embedding a closing 👉 question, and the `ask-bootcamper` hook SHALL generate the appropriate contextual question when the agent stops

2.3 WHEN the `ask-bootcamper` hook fires after the agent completes a module step THEN the hook SHALL be the single source that generates a contextual 👉 question based on what the agent just presented, eliminating the possibility of dropped prefixes, duplicate questions, or ownership conflicts

2.4 WHEN a module steering file contains a mandatory offer (e.g., visualization offers in modules 06, 07, 08 marked with "DO NOT SKIP") THEN the steering file SHALL instruct the agent to present the offer as informational content without an inline 👉 closing question or WAIT instruction, and the hook SHALL generate the closing question

### Unchanged Behavior (Regression Prevention)

3.1 WHEN the `ask-bootcamper` hook fires on `agentStop` THEN the system SHALL CONTINUE TO generate contextual 👉 questions based on what the agent just accomplished

3.2 WHEN the `agent-instructions.md` closing-question ownership rule is evaluated THEN the system SHALL CONTINUE TO enforce that the `ask-bootcamper` hook owns all closing questions

3.3 WHEN the agent presents informational content within a module step (explanations, tables, code examples, MCP query results) THEN the system SHALL CONTINUE TO present that content unchanged — only trailing inline 👉 questions and WAIT instructions are removed

3.4 WHEN the agent executes module steps THEN the system SHALL CONTINUE TO follow the same step sequence, checkpoint logic, and decision gates defined in each module steering file

3.5 WHEN the agent executes the onboarding flow THEN the system SHALL CONTINUE TO use the already-fixed `onboarding-flow.md` without regression — the fix applied in `.kiro/specs/missing-pointing-prefix/` SHALL remain intact

3.6 WHEN the `ask-bootcamper` hook file or `agent-instructions.md` file is evaluated THEN the system SHALL CONTINUE TO use these files unmodified — the fix is scoped to module steering files only
