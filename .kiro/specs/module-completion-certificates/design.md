# Design: Module Completion Certificates

## Overview

When a module is marked complete, the agent generates a Markdown certificate documenting what was built, concepts learned, and artifacts produced. Certificates accumulate in `docs/progress/` and are included in exports.

## Certificate Template

```markdown
# Module [N]: [Title] — Complete ✅

**Completed**: [ISO 8601 date]
**Time Spent**: [duration or "not tracked"]
**Language**: [chosen language]

## Key Concepts Learned

- [Concept 1 — one sentence]
- [Concept 2 — one sentence]
- [Concept 3 — one sentence]
- [Concept 4 — one sentence (if applicable)]
- [Concept 5 — one sentence (if applicable)]

## Artifacts Produced

| File | Description |
|------|-------------|
| `src/load/load.py` | Data loading script for [sources] |
| `config/mapping_spec.json` | Senzing mapping specification |

## What This Enables

Now that you've completed Module [N], you can:
- [Capability 1]
- [Capability 2]
- [Capability 3]

## Session Stats

| Metric | Value |
|--------|-------|
| Total turns | [N] |
| Corrections | [N] |
| Time per step (avg) | [duration] |

---
*Generated by Senzing Bootcamp Power*
```

The "Session Stats" section is only included if session analytics are available.

## Content Derivation

Certificates are NOT generic boilerplate. Content comes from:

1. **Key Concepts**: Derived from the module's steering file section headings and step descriptions. Each module has 3-5 core concepts that the steering file teaches.
2. **Artifacts**: Read from the actual file system — list files in the module's output directories that were created/modified during the module.
3. **What This Enables**: Derived from `module-dependencies.yaml` — what modules are now unlocked, and what capabilities those represent.
4. **Language**: From `config/bootcamp_preferences.yaml` — used in artifact descriptions ("Built a Python loading script" not "Built a loading script").

## Module Concept Mapping

The agent derives concepts from the module steering file, but here's the expected mapping:

| Module | Core Concepts |
|--------|--------------|

## Summary Index

`docs/progress/README.md` is maintained as a living index:

```markdown
# Bootcamp Progress

## Completed Modules

| Module | Title | Completed | Time |
|--------|-------|-----------|------|
| 1 | Business Problem | 2026-05-01 | 45 min |
| 2 | SDK Setup | 2026-05-02 | 30 min |

## Track Progress

**Fast Track (B)**: 2/3 modules complete (67%)

## Total Time Invested

1 hour 15 minutes across 2 modules
```

## Integration with module-completion.md

The certificate generation step is added to the module-completion workflow after the completion celebration and before the next-module offer:

```console
1. Mark module complete in progress.json ✓
2. Display completion celebration ✓
3. Generate certificate → docs/progress/MODULE_N_COMPLETE.md  [NEW]
4. Update summary index → docs/progress/README.md  [NEW]
5. Offer next module / export / graduation
```

## Integration with export_results.py

The export script checks for `docs/progress/` and includes certificates:

- HTML export: "Achievements" section with certificate content
- ZIP export: `artifacts/progress/` directory with all certificate files

## Files Created/Modified

- `senzing-bootcamp/steering/module-completion.md` — add certificate generation step
- `senzing-bootcamp/scripts/export_results.py` — include certificates in export
- Agent creates at runtime: `docs/progress/MODULE_N_COMPLETE.md`, `docs/progress/README.md`

## Testing

- Unit test: module-completion.md contains certificate generation instruction
- Unit test: certificate template contains all required sections
- Property test: certificate file naming follows MODULE_N_COMPLETE.md pattern for all valid N
- Unit test: export_results.py handles presence/absence of docs/progress/ gracefully
