# Requirements Document

## Introduction

The Senzing Bootcamp feedback workflow collects improvement suggestions into a markdown file (`SENZING_BOOTCAMP_POWER_FEEDBACK.md`) using a structured template with fields for date, module, priority, category, description, and suggested fix. Currently, turning this feedback into actionable specs requires a maintainer to manually read each entry, classify it, create a spec directory, and populate the initial document. This feature adds a triage script (`scripts/triage_feedback.py`) that parses the feedback markdown file and generates skeleton spec directories with pre-populated `bugfix.md` or `requirements.md` files, reducing the manual effort from minutes per feedback item to a single command invocation.

## Glossary

- **Triage_Script**: The `scripts/triage_feedback.py` Python script that parses feedback and generates skeleton spec directories
- **Feedback_File**: A markdown file following the `SENZING_BOOTCAMP_POWER_FEEDBACK_TEMPLATE.md` structure, containing one or more Feedback_Entries
- **Feedback_Entry**: A single improvement suggestion in the Feedback_File, delimited by a level-2 heading matching the pattern `## Improvement: <title>`
- **Feedback_Category**: The category field of a Feedback_Entry, one of: Documentation, Workflow, Tools, UX, Bug, Performance, Security
- **Feedback_Priority**: The priority field of a Feedback_Entry, one of: High, Medium, Low
- **Spec_Directory**: A directory under `.kiro/specs/` containing spec documents for a single feature or bugfix
- **Skeleton_Bugfix**: A pre-populated `bugfix.md` file generated by the Triage_Script for Feedback_Entries with category "Bug"
- **Skeleton_Requirements**: A pre-populated `requirements.md` file generated by the Triage_Script for Feedback_Entries with non-bug categories
- **Config_File**: The `.config.kiro` JSON file created in each Spec_Directory to register the spec with the workflow system
- **Triage_Report**: A summary output printed by the Triage_Script listing all generated Spec_Directories and any Feedback_Entries that could not be parsed

## Requirements

### Requirement 1: Feedback File Parsing

**User Story:** As a power maintainer, I want the triage script to parse the feedback markdown file into structured entries, so that each piece of feedback can be processed independently.

#### Acceptance Criteria

1. THE Triage_Script SHALL accept a file path argument specifying the Feedback_File to parse
2. WHEN no file path argument is provided, THE Triage_Script SHALL default to `SENZING_BOOTCAMP_POWER_FEEDBACK.md` in the current working directory
3. THE Triage_Script SHALL identify each Feedback_Entry by scanning for level-2 headings matching the pattern `## Improvement: <title>`
4. THE Triage_Script SHALL extract the following fields from each Feedback_Entry: title (from the heading), date, module number, priority, category, "What Happened" content, "Why It's a Problem" content, "Suggested Fix" content, and "Workaround Used" content
5. WHEN a Feedback_Entry is missing one or more required fields (title, category), THE Triage_Script SHALL log a warning identifying the entry and the missing fields and skip that entry
6. WHEN a Feedback_Entry contains an unrecognized Feedback_Category, THE Triage_Script SHALL log a warning and default to generating a Skeleton_Requirements document for that entry

### Requirement 2: Spec Directory Generation

**User Story:** As a power maintainer, I want the triage script to create properly structured spec directories, so that generated specs integrate with the existing workflow system.

#### Acceptance Criteria

1. THE Triage_Script SHALL create a Spec_Directory under `.kiro/specs/` for each successfully parsed Feedback_Entry
2. THE Triage_Script SHALL derive the Spec_Directory name from the Feedback_Entry title by converting it to kebab-case (lowercase, spaces and special characters replaced with hyphens, consecutive hyphens collapsed)
3. WHEN a Spec_Directory with the derived name already exists, THE Triage_Script SHALL skip that entry and log a warning indicating the directory already exists
4. THE Triage_Script SHALL create a Config_File in each Spec_Directory with the JSON structure `{"specId": "<uuid>", "workflowType": "<type>", "specType": "<spec_type>"}` where the specId is a unique UUID v4, workflowType is `"requirements-first"` for non-bug categories and `"bugfix"` for bug categories, and specType is `"feature"` for non-bug categories and `"bugfix"` for bug categories
5. THE Triage_Script SHALL generate a unique UUID v4 for each Config_File specId

### Requirement 3: Bug Category Skeleton Generation

**User Story:** As a power maintainer, I want bug feedback to generate a bugfix spec skeleton, so that the bug investigation workflow starts with the reported context already filled in.

#### Acceptance Criteria

1. WHEN a Feedback_Entry has Feedback_Category "Bug", THE Triage_Script SHALL generate a Skeleton_Bugfix file named `bugfix.md` in the Spec_Directory
2. THE Skeleton_Bugfix SHALL contain a "Bug Report" section populated with the "What Happened" content from the Feedback_Entry
3. THE Skeleton_Bugfix SHALL contain a "Steps to Reproduce" section populated with contextual information extracted from the Feedback_Entry (module number, date, and any sequential steps described in the "What Happened" content)
4. THE Skeleton_Bugfix SHALL contain an "Expected Behavior" section derived from the inverse of the "Why It's a Problem" content
5. THE Skeleton_Bugfix SHALL contain a "Suggested Fix" section populated with the "Suggested Fix" content from the Feedback_Entry
6. WHEN the Feedback_Entry includes a "Workaround Used" section with content, THE Skeleton_Bugfix SHALL include a "Known Workaround" section with that content

### Requirement 4: Non-Bug Category Skeleton Generation

**User Story:** As a power maintainer, I want non-bug feedback to generate a requirements spec skeleton, so that feature and improvement work starts with the user's context already captured.

#### Acceptance Criteria

1. WHEN a Feedback_Entry has a Feedback_Category other than "Bug", THE Triage_Script SHALL generate a Skeleton_Requirements file named `requirements.md` in the Spec_Directory
2. THE Skeleton_Requirements SHALL contain an "Introduction" section summarizing the Feedback_Entry using the title, "What Happened" content, and "Why It's a Problem" content
3. THE Skeleton_Requirements SHALL contain a "Glossary" section with placeholder entries derived from key terms in the Feedback_Entry title
4. THE Skeleton_Requirements SHALL contain a "Requirements" section with one initial requirement stub that includes a user story derived from the "Suggested Fix" content and empty acceptance criteria placeholders
5. THE Skeleton_Requirements SHALL include a comment at the top indicating it was auto-generated by the Triage_Script and requires human review before implementation

### Requirement 5: Triage Report Output

**User Story:** As a power maintainer, I want a summary report after triage, so that I can see what was generated and what needs manual attention.

#### Acceptance Criteria

1. THE Triage_Script SHALL print a Triage_Report to stdout after processing all Feedback_Entries
2. THE Triage_Report SHALL list each generated Spec_Directory with its path, the source Feedback_Entry title, the generated document type (bugfix or requirements), and the Feedback_Priority
3. THE Triage_Report SHALL list any Feedback_Entries that were skipped due to parsing errors, missing fields, or pre-existing directories, with the reason for skipping
4. THE Triage_Report SHALL print a summary line showing the total number of entries processed, specs generated, and entries skipped
5. WHEN zero Feedback_Entries are found in the Feedback_File, THE Triage_Script SHALL print a message indicating no improvement entries were found and exit with exit code 0

### Requirement 6: CLI Interface and Error Handling

**User Story:** As a CI pipeline maintainer, I want the triage script to have a predictable CLI interface, so that it can be integrated into automation workflows.

#### Acceptance Criteria

1. THE Triage_Script SHALL run as `python scripts/triage_feedback.py [path]` from the repository root with no third-party dependencies beyond the Python standard library
2. WHEN the specified Feedback_File does not exist, THE Triage_Script SHALL print an error message identifying the missing file and exit with exit code 1
3. WHEN the Triage_Script successfully generates one or more Spec_Directories, THE Triage_Script SHALL exit with exit code 0
4. WHEN the Triage_Script encounters a file system error while creating a Spec_Directory or writing a file, THE Triage_Script SHALL log the error, skip that entry, and continue processing remaining entries
5. THE Triage_Script SHALL support a `--dry-run` flag that prints the Triage_Report showing what would be generated without creating any files or directories
6. THE Triage_Script SHALL support a `--output-dir` flag that overrides the default `.kiro/specs/` base directory for generated Spec_Directories

### Requirement 7: Feedback Entry Parser Round-Trip Integrity

**User Story:** As a power maintainer, I want confidence that the parser extracts feedback content faithfully, so that no user-reported context is lost or corrupted during triage.

#### Acceptance Criteria

1. THE Triage_Script parser SHALL preserve all markdown formatting (bold, italic, code blocks, lists) within extracted Feedback_Entry field content
2. THE Triage_Script parser SHALL handle Feedback_Entries that contain multiple paragraphs within a single field section
3. THE Triage_Script parser SHALL handle Feedback_Entries where optional fields ("Workaround Used") are present but empty
4. FOR ALL valid Feedback_Entries, parsing the entry and writing it back to the skeleton document SHALL preserve the substantive content of the "What Happened", "Why It's a Problem", and "Suggested Fix" fields without truncation or alteration
