# Requirements Document

## Introduction

The senzing-bootcamp power's `status.py` script currently outputs bootcamp progress, project health, and next steps as plain text to the terminal. While functional, this output is not visually engaging, not easily shareable, and lacks the richer data visualizations that would help bootcampers and stakeholders understand progress at a glance. This feature extends `status.py` with an `--html` flag that generates a self-contained HTML dashboard showing module completion progress, data quality scores, performance metrics, entity resolution statistics, a module completion timeline, and project health checks — then auto-opens the dashboard in the default browser. The HTML file uses inline CSS and JavaScript with no external dependencies, making it portable and shareable.

## Glossary

- **Status_Script**: The Python script at `senzing-bootcamp/scripts/status.py` that reports bootcamp progress and project health.
- **Dashboard**: A single self-contained HTML file generated by the Status_Script when invoked with the `--html` flag, displaying bootcamp progress and metrics in a visual layout.
- **Progress_Data**: The JSON structure stored in `config/bootcamp_progress.json` that tracks module completion state, current module, language choice, and data sources.
- **Progress_Tracker**: The markdown file at `docs/guides/PROGRESS_TRACKER.md` used as a fallback when Progress_Data is unavailable.
- **Quality_Score**: The 0–100 weighted score produced during Module 5 that measures data completeness, consistency, format compliance, and uniqueness.
- **Quality_Band**: A color-coded classification of a Quality_Score: green (≥ 80), yellow (70–79), or red (< 70).
- **Performance_Metrics**: Throughput and timing measurements produced during Modules 6, 7, and 9 — including records-per-second, wall-clock loading time, and query response times.
- **Entity_Statistics**: Entity resolution outcome data produced during Modules 6–8 — including total entities resolved, match counts, cross-source matches, and duplicate counts.
- **Module_Card**: A visual element in the Dashboard representing a single bootcamp module, showing its name, number, and completion status.
- **Progress_Bar**: A horizontal visual element in the Dashboard that fills proportionally to the percentage of completed modules.
- **Timeline**: A chronological visual element in the Dashboard showing when each module was completed.
- **Health_Check**: A diagnostic check that verifies the existence of expected project directories and files (data, database, src, scripts, .gitignore, .env.example, README.md, backups).

## Requirements

### Requirement 1: HTML Flag and Script Integration

**User Story:** As a bootcamper, I want to run `status.py --html` to generate a visual dashboard, so that I can see my progress in a richer format than terminal text.

#### Acceptance Criteria

1. THE Status_Script SHALL accept an `--html` flag as a command-line argument.
2. WHEN the `--html` flag is provided, THE Status_Script SHALL generate a Dashboard file instead of printing terminal output.
3. WHEN the `--html` flag is not provided, THE Status_Script SHALL produce the same terminal output as the current implementation.
4. THE Status_Script SHALL accept an optional `--output` argument specifying the Dashboard output file path, defaulting to `docs/dashboard.html`.
5. IF the output directory for the Dashboard does not exist, THEN THE Status_Script SHALL create the output directory before writing the file.
6. WHEN the Dashboard file is written, THE Status_Script SHALL print the output file path and file size to the terminal.

### Requirement 2: Self-Contained HTML Output

**User Story:** As a bootcamper, I want the dashboard to be a single HTML file with no external dependencies, so that I can share it by email or open it on any machine.

#### Acceptance Criteria

1. THE Dashboard SHALL embed all CSS styles inline within a `<style>` tag in the HTML `<head>`.
2. THE Dashboard SHALL embed all JavaScript inline within `<script>` tags in the HTML `<body>`.
3. THE Dashboard SHALL not reference any external stylesheets, scripts, fonts, or images.
4. THE Dashboard SHALL use semantic HTML elements (`<header>`, `<main>`, `<section>`, `<footer>`, `<nav>`) for structure.
5. THE Dashboard SHALL be viewable in any modern web browser (Chrome, Firefox, Safari, Edge) without requiring a web server.
6. THE Dashboard SHALL include a generation timestamp in the footer.
7. THE Status_Script SHALL depend only on the Python standard library for Dashboard generation (no third-party packages).

### Requirement 3: Auto-Open in Browser

**User Story:** As a bootcamper, I want the dashboard to open automatically in my browser after generation, so that I can see results immediately without navigating to the file.

#### Acceptance Criteria

1. WHEN the Dashboard file is generated, THE Status_Script SHALL open the Dashboard in the default web browser using the Python `webbrowser` module.
2. THE Status_Script SHALL accept a `--no-open` flag that suppresses automatic browser opening.
3. IF the browser fails to open, THEN THE Status_Script SHALL print the file path so the bootcamper can open the Dashboard manually.

### Requirement 4: Module Completion Progress Display

**User Story:** As a bootcamper, I want to see my module completion progress with a visual progress bar and module cards, so that I can quickly understand how far I have come.

#### Acceptance Criteria

1. THE Dashboard SHALL display a Progress_Bar showing the percentage of completed modules out of 12 total.
2. THE Dashboard SHALL display the completion count as a fraction (e.g., "7 / 12 modules").
3. THE Dashboard SHALL display a Module_Card for each of the 12 bootcamp modules.
4. WHEN a module is completed, THE Module_Card SHALL display a completed visual indicator.
5. WHEN a module is in progress, THE Module_Card SHALL display an in-progress visual indicator.
6. WHEN a module is not started, THE Module_Card SHALL display a not-started visual indicator.
7. THE Dashboard SHALL display the chosen programming language from the Progress_Data when available.
8. THE Dashboard SHALL display the current bootcamp status (Not Started, In Progress, Ready to Start, or Complete).

### Requirement 5: Data Quality Scores Display

**User Story:** As a bootcamper, I want to see my data quality scores with color-coded bands, so that I can quickly assess whether my data is ready for entity resolution.

#### Acceptance Criteria

1. WHEN quality assessment data exists in project artifacts (quality score reports from Module 5), THE Dashboard SHALL display a quality scores section.
2. THE quality scores section SHALL display the overall Quality_Score for each data source as a numeric value (0–100).
3. THE quality scores section SHALL display the Quality_Band color for each overall score: green for scores ≥ 80, yellow for scores 70–79, red for scores < 70.
4. THE quality scores section SHALL display sub-scores for completeness, consistency, format compliance, and uniqueness when available.
5. WHEN quality assessment data is not available, THE Dashboard SHALL omit the quality scores section entirely rather than displaying empty content.

### Requirement 6: Performance Metrics Display

**User Story:** As a bootcamper, I want to see my performance metrics (records/sec, query times), so that I can understand how my setup performs.

#### Acceptance Criteria

1. WHEN Performance_Metrics exist in project artifacts (from Modules 6, 7, or 9), THE Dashboard SHALL display a performance metrics section.
2. THE performance metrics section SHALL display loading throughput in records per second when available.
3. THE performance metrics section SHALL display query response times (average and P95) when available.
4. THE performance metrics section SHALL display the database type (SQLite or PostgreSQL) when available.
5. THE performance metrics section SHALL display wall-clock loading time when available.
6. WHEN Performance_Metrics are not available, THE Dashboard SHALL omit the performance metrics section entirely rather than displaying empty content.

### Requirement 7: Entity Resolution Statistics Display

**User Story:** As a bootcamper, I want to see entity resolution statistics (entities resolved, matches, duplicates), so that I can understand the outcomes of my entity resolution work.

#### Acceptance Criteria

1. WHEN Entity_Statistics exist in project artifacts (from Modules 6–8), THE Dashboard SHALL display an entity resolution statistics section.
2. THE entity resolution statistics section SHALL display total records loaded when available.
3. THE entity resolution statistics section SHALL display total entities resolved when available.
4. THE entity resolution statistics section SHALL display match counts when available.
5. THE entity resolution statistics section SHALL display duplicate counts when available.
6. THE entity resolution statistics section SHALL display cross-source match counts when available.
7. WHEN Entity_Statistics are not available, THE Dashboard SHALL omit the entity resolution statistics section entirely rather than displaying empty content.

### Requirement 8: Module Completion Timeline

**User Story:** As a bootcamper, I want to see a timeline of when I completed each module, so that I can track my learning pace and share my journey.

#### Acceptance Criteria

1. WHEN module completion timestamps are available in the Progress_Data, THE Dashboard SHALL display a Timeline section showing completed modules in chronological order.
2. THE Timeline SHALL display each completed module with its module number, name, and completion date.
3. THE Timeline SHALL present modules in chronological order based on completion date.
4. WHEN module completion timestamps are not available in the Progress_Data, THE Dashboard SHALL omit the Timeline section rather than displaying empty content.

### Requirement 9: Project Health Checks Display

**User Story:** As a bootcamper, I want the dashboard to include the same project health checks as the terminal output, so that I can see project structure status in the visual format.

#### Acceptance Criteria

1. THE Dashboard SHALL display a project health section showing the same Health_Check items as the terminal output: data directory, database directory, source directory, scripts directory, .gitignore, .env.example, README.md, and backups directory.
2. THE Dashboard SHALL display each Health_Check item with a pass or fail indicator based on whether the path exists.
3. THE Dashboard SHALL display an overall health score as a fraction and percentage (e.g., "6/8 (75%)").

### Requirement 10: Data Source Handling

**User Story:** As a bootcamper, I want the dashboard to read the same data sources as the terminal status command, so that the dashboard is consistent with what I see in the terminal.

#### Acceptance Criteria

1. THE Status_Script SHALL read Progress_Data from `config/bootcamp_progress.json` as the primary data source for the Dashboard.
2. WHEN `config/bootcamp_progress.json` does not exist, THE Status_Script SHALL fall back to parsing `docs/guides/PROGRESS_TRACKER.md` for module completion data.
3. WHEN neither Progress_Data nor Progress_Tracker exists, THE Dashboard SHALL display a notice that no progress data was found and show only the project health section.
4. THE Status_Script SHALL scan project artifact files (quality reports, performance logs, entity statistics) to populate the quality, performance, and entity resolution sections.
5. IF a file read error occurs for any individual data source, THEN THE Status_Script SHALL skip that data source and continue generating the Dashboard with available data.

### Requirement 11: Error Handling

**User Story:** As a bootcamper, I want clear feedback when something goes wrong during dashboard generation, so that I can troubleshoot issues.

#### Acceptance Criteria

1. IF the Status_Script cannot write the Dashboard file due to a permission or disk error, THEN THE Status_Script SHALL print a descriptive error message and exit with a non-zero exit code.
2. IF the Progress_Data file contains invalid JSON, THEN THE Status_Script SHALL print a warning, skip the corrupted data, and generate the Dashboard with whatever data is available.
3. THE Status_Script SHALL not raise unhandled exceptions during Dashboard generation; all errors SHALL be caught and reported to the terminal.

### Requirement 12: Cross-Platform Compatibility

**User Story:** As a bootcamper, I want the dashboard generation to work on Linux, macOS, and Windows, so that the feature works regardless of my operating system.

#### Acceptance Criteria

1. THE Status_Script SHALL use platform-independent path handling (via `pathlib`) for all file operations during Dashboard generation.
2. THE Status_Script SHALL use the Python `webbrowser` module for cross-platform browser opening.
3. THE Dashboard HTML output SHALL use UTF-8 encoding.
