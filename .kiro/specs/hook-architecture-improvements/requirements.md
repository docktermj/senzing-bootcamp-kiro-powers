# Requirements Document

## Introduction

This feature addresses three related problems in the hook architecture of the `senzing-bootcamp`
Kiro Power. All three concern how hooks are ordered, authored, and installed — and all three are
about making hook behavior deterministic, maintainable, and reliably present so the bootcamp's
guarantees (single-question protocol, mandatory gates, journey fidelity) hold up in practice.

The three themes are:

- **Theme A — agentStop hook ordering and documentation.** Multiple hooks fire on `agentStop`,
  each reading progress files and running prompt logic at every turn end. There is no documented,
  deterministic intended fire order, and the combined output behavior is not characterized. This
  theme produces an authoritative, documented precedence order and per-hook guard expectations so
  the effective behavior is deterministic without requiring control over the IDE's execution engine.

- **Theme B — hook-prompt deduplication.** The Module 3 gate logic (the "is this a Module 3
  completion write" check, CONDITION A checkpoint checks for `web_service`/`web_page`, and the ⛔
  output strings) is duplicated nearly verbatim across three hook files and mirrored again in a
  steering registry document. This verbatim duplication is how an escape hatch has historically
  hidden in one copy. This theme introduces a build-time single-source mechanism for shared hook
  prompt fragments, plus CI verification that authored hook JSON has not drifted from its composed
  source.

- **Theme C — capture-hook install reliability.** Completion-summary and journey-fidelity behavior
  depends on the capture hooks (`session-log-events`, `module-recap-append`) and the closing-question
  owner (`ask-bootcamper`) being installed, but installation is opt-in through an interactive
  installer whose curated hook list is stale (it references three consolidated hooks that no longer
  exist and omits several current hooks). This theme repairs the installer, adds a non-interactive
  mode, guarantees capture-critical hooks are covered across both install paths, and adds tests that
  prevent the curated list from silently rotting again.

### Grounded facts and scope corrections

These facts were verified against the repository and govern the requirements below.

- **agentStop hooks — exactly FIVE, not six.** `hooks.lock.yaml` and the hook JSON files confirm the
  hooks whose `when.type` is `agentStop` are: `ask-bootcamper`, `enforce-gate-on-stop`,
  `enforce-visualization-offers`, `module-completion-celebration`, and `module-recap-append`.
  `session-log-events` is a `postToolUse` hook (`when.type: postToolUse`, `toolTypes: ["write"]`),
  **not** an `agentStop` hook. The originating improvement note listed six agentStop hooks including
  `session-log-events`; that count is corrected here. Theme A applies to the five agentStop hooks.
  `session-log-events` remains capture-critical for Theme C regardless of its event type.

- **The power cannot control IDE-level hook firing order.** The IDE hook-execution engine decides
  the actual order in which `agentStop` hooks fire, and this feature does **not** modify that engine.
  Determinism of *effective* behavior is achieved by (a) per-hook guard conditions that make each
  hook a silent no-op unless its specific condition holds, and (b) a documented precedence rule that
  resolves the rare case where more than one hook would emit output in the same turn.

- **`.kiro.hook` `then.prompt` must remain a self-contained string at runtime.** The IDE reads each
  hook JSON as-is; there is no include/expansion mechanism at hook-execution time. Therefore the
  "single source" for shared fragments in Theme B must be a **build/generation-time** mechanism that
  composes the final self-contained prompt string into the hook JSON, plus a CI `--verify` mode that
  detects drift — analogous to the existing `sync_hook_registry.py --verify`.

- **Two install paths exist.** Capture hooks reach a bootcamper's project through either (1) the
  `createHook`-from-registry path that the agent runs during onboarding/module start (driven by
  `agent-instructions.md` and the `hook-registry-*.md` files generated by `sync_hook_registry.py`),
  or (2) the `install_hooks.py` file-copy path the bootcamper may run manually. Theme C must address
  capture-hook coverage on **both** paths.

- **Security constraint.** `preToolUse` hooks with `toolTypes: ["write"]` (e.g.,
  `gate-module3-visualization`, `enforce-mandatory-gate`, `write-policy-gate`) must not be removed.
  Theme B's deduplication is a refactor that preserves their runtime behavior byte-for-byte.

## Glossary

- **Hook**: A `.kiro.hook` JSON file with fields `name`, `version`, `when`, `then`. The `when` block
  defines the triggering event (`when.type`) and optional filters (`toolTypes`, `patterns`); the
  `then` block defines the action (`then.type`) and, for `askAgent` hooks, the prompt string
  (`then.prompt`).
- **agentStop hook**: A hook with `when.type` equal to `agentStop`. There are exactly five in the
  power (see grounded facts).
- **Capture hook**: A hook whose purpose is to record session/journey data for the completion
  summary and recap: `session-log-events` (`postToolUse`) and `module-recap-append` (`agentStop`).
- **Closing-question owner**: The `ask-bootcamper` hook, which owns end-of-turn closing questions and
  the answer-processing retry, per `agent-instructions.md`.
- **Guard condition**: The opening conditional logic in a hook's prompt that causes the hook to
  produce zero output ("no-op") unless a specific state holds (e.g., "if `current_module` is not 3,
  produce no output").
- **Question-pending state**: The state in which `config/.question_pending` exists and/or the most
  recent assistant message contains a 👉 question awaiting a bootcamper response. Per
  `agent-instructions.md`, agentStop hooks must emit zero output in this state.
- **Hook prompt fragment**: A named, reusable unit of hook prompt text (e.g., the Module 3 CONDITION
  A checkpoint check) defined once and composed into one or more hook prompts at build time.
- **Fragment source**: The authoritative file(s) defining hook prompt fragments and which hooks
  reference which fragments.
- **Prompt composer**: The build-time tool that expands fragment references into the concrete,
  self-contained `then.prompt` strings inside the `.kiro.hook` files.
- **Curated hook list**: The hardcoded `HOOKS` list and `ESSENTIAL` set inside `install_hooks.py`.
- **Critical hooks**: The hooks listed under `critical` in `hook-categories.yaml`
  (`ask-bootcamper`, `code-style-check`, `commonmark-validation`, `review-bootcamper-input`,
  `write-policy-gate`) — created during onboarding.
- **Consolidated hooks**: `enforce-file-path-policies`, `enforce-single-question`, and
  `block-direct-sql`, which were merged into `write-policy-gate` and no longer exist as files.
- **CI verification suite**: The validators run in `.github/workflows/validate-power.yml`:
  `validate_power.py`, `measure_steering.py --check`, `validate_commonmark.py`,
  `sync_hook_registry.py --verify`, and pytest.

## Requirements

## Theme A — agentStop Hook Ordering and Documentation

### Requirement 1: [Theme A] Authoritative documented fire order for agentStop hooks

**User Story:** As a power maintainer, I want a single authoritative, documented intended fire order
for the agentStop hooks, so that the combined end-of-turn behavior is predictable and reviewable.

#### Acceptance Criteria

1. THE Hook_Architecture_Documentation SHALL record an ordered precedence list of exactly the five
   agentStop hooks: `ask-bootcamper`, `module-recap-append`, `module-completion-celebration`,
   `enforce-gate-on-stop`, and `enforce-visualization-offers`.
2. THE Hook_Architecture_Documentation SHALL define the intended precedence as: (1) answer-processing
   and closing-question ownership (`ask-bootcamper`), (2) capture (`module-recap-append`),
   (3) celebration (`module-completion-celebration`), (4) gate enforcement (`enforce-gate-on-stop`),
   (5) visualization-offer enforcement (`enforce-visualization-offers`).
3. THE Hook_Architecture_Documentation SHALL record a written rationale for the relative position of
   each agentStop hook in the precedence list.
4. WHERE the precedence order is recorded, THE Hook_Architecture_Documentation SHALL store it in a
   machine-readable location (an `agentstop_order` mapping in `hook-categories.yaml`) and reference
   it from a steering document.
5. THE `agentstop_order` mapping SHALL list every hook whose `when.type` is `agentStop` in the hook
   directory, and SHALL list no hook whose `when.type` is not `agentStop`.

### Requirement 2: [Theme A] Ordering is consistent with closing-question ownership and question-pending silence

**User Story:** As a bootcamper, I want no agentStop hook to interfere with a pending question, so
that the turn ends cleanly when I still owe an answer.

#### Acceptance Criteria

1. WHILE the Question-pending state holds, THE agentStop_Hooks SHALL each produce zero visible output.
2. THE Hook_Architecture_Documentation SHALL state that `ask-bootcamper` is the closing-question
   owner and that all other agentStop hooks defer closing-question emission to `ask-bootcamper`.
3. IF the documented precedence order conflicts with the existing rule that `ask-bootcamper` owns
   closing questions, THEN THE Hook_Architecture_Documentation SHALL resolve the conflict in favor of
   `ask-bootcamper` owning closing questions.
4. WHERE a hook prompt's guard condition is inspected, THE agentStop_Hooks SHALL each contain guard
   text that yields zero output when `config/.question_pending` exists.

### Requirement 3: [Theme A] Precedence and silence rules when multiple agentStop hooks would emit output

**User Story:** As a bootcamper, I want at most one coherent end-of-turn message instead of stacked
hook outputs, so that the conversation stays readable.

#### Acceptance Criteria

1. WHEN more than one agentStop hook would each produce visible output in the same turn, THE
   Hook_Architecture_Documentation SHALL define which single hook's output takes precedence according
   to the ordered list in Requirement 1.
2. THE Hook_Architecture_Documentation SHALL specify that a gate-violation output from
   `enforce-gate-on-stop` takes precedence over a celebration output from
   `module-completion-celebration` in the same turn.
3. THE Hook_Architecture_Documentation SHALL specify that no agentStop hook appends its output after
   another agentStop hook's output in a way that stacks two separate end-of-turn messages.
4. WHEN no agentStop hook's guard condition is satisfied in a turn, THE agentStop_Hooks SHALL
   collectively produce zero visible output.

### Requirement 4: [Theme A] Per-hook guard determinism (no IDE engine change)

**User Story:** As a power maintainer, I want effective ordering to be guaranteed by per-hook guards
rather than by controlling the IDE, so that determinism does not depend on engine internals.

#### Acceptance Criteria

1. THE Hook_Architecture_Improvements SHALL NOT modify the IDE hook-execution engine or the order in
   which the IDE dispatches agentStop hooks.
2. THE Hook_Architecture_Documentation SHALL state the assumption that the power cannot control
   IDE-level firing order and that determinism is achieved through per-hook guard conditions plus the
   documented precedence rule.
3. WHERE an agentStop hook's guard condition is not satisfied, THE agentStop_Hook SHALL produce zero
   visible output regardless of the order in which the IDE fired it.
4. THE agentStop hooks' guard conditions SHALL be mutually consistent such that, for any given
   progress state, at most one hook emits a primary end-of-turn message.

## Theme B — Hook-Prompt Deduplication

### Requirement 5: [Theme B] Single-source definition of shared hook-prompt fragments

**User Story:** As a power maintainer, I want shared gate logic authored exactly once, so that fixing
or auditing it does not require editing multiple copies.

#### Acceptance Criteria

1. THE Fragment_Source SHALL define the shared Module 3 gate logic — the CONDITION A checkpoint check
   (`module_3_verification.checks.web_service.status` equals `"passed"` and
   `module_3_verification.checks.web_page.status` equals `"passed"`) — as a single named fragment.
2. THE Fragment_Source SHALL define each shared ⛔ output string used by the Module 3 gate hooks as a
   named fragment.
3. THE Fragment_Source SHALL be the only authoritative location where each shared fragment's text is
   defined.
4. WHERE a hook reuses a shared fragment, THE hook definition SHALL reference the fragment by name in
   the Fragment_Source rather than restating the fragment text in a separate authored copy.
5. THE Fragment_Source SHALL be a Python 3.11+ stdlib-only artifact or a data file parseable without
   third-party dependencies, consistent with the repository's no-dependency constraint.

### Requirement 6: [Theme B] Build-time composition into self-contained hook prompts

**User Story:** As a power maintainer, I want a generator that expands fragment references into the
final hook JSON, so that runtime hook prompts remain self-contained strings the IDE can read as-is.

#### Acceptance Criteria

1. THE Prompt_Composer SHALL produce, for each hook that references fragments, a concrete
   `then.prompt` string in which all fragment references have been expanded to their full text.
2. THE Prompt_Composer SHALL produce hook JSON files in which `then.prompt` is a single self-contained
   string containing no unexpanded fragment-reference markers.
3. WHEN the Prompt_Composer runs in write mode, THE Prompt_Composer SHALL write the composed
   `.kiro.hook` files to the hooks directory.
4. THE Prompt_Composer SHALL be runnable from the command line with a `main(argv=None)` entry point,
   argparse-based flags, and exit code 0 on success and 1 on error.
5. WHEN the Prompt_Composer encounters a fragment reference that has no definition in the
   Fragment_Source, THEN THE Prompt_Composer SHALL exit with code 1 and report the unknown fragment
   name.

### Requirement 7: [Theme B] CI-verifiable drift detection for composed hook prompts

**User Story:** As a power maintainer, I want CI to fail if an authored hook prompt drifts from its
composed source, so that the duplication problem cannot silently re-emerge.

#### Acceptance Criteria

1. THE Prompt_Composer SHALL provide a `--verify` mode that compares the on-disk `.kiro.hook` files
   against the content the composer would generate from the Fragment_Source.
2. WHEN every composed hook prompt matches its on-disk `.kiro.hook` file, THE Prompt_Composer
   `--verify` mode SHALL exit with code 0.
3. IF any on-disk `.kiro.hook` prompt differs from the composer's generated content, THEN THE
   Prompt_Composer `--verify` mode SHALL exit with code 1 and report which hook drifted.
4. THE CI_Verification_Suite SHALL invoke the Prompt_Composer `--verify` mode as a required step.
5. WHEN two hooks reference the same shared fragment, THE Prompt_Composer SHALL expand byte-identical
   text for that fragment into both hooks' prompts.

### Requirement 8: [Theme B] Behavior and schema preservation across deduplication

**User Story:** As a bootcamper, I want the deduplication to change nothing about how the gate hooks
behave, so that the Module 3 mandatory gate keeps working exactly as before.

#### Acceptance Criteria

1. WHEN the Prompt_Composer composes the Module 3 gate hooks
   (`gate-module3-visualization`, `enforce-mandatory-gate`, `enforce-gate-on-stop`), THE composed
   `then.prompt` strings SHALL be semantically equivalent to the prompts present before
   deduplication.
2. THE composed `.kiro.hook` files SHALL each contain valid JSON with the required fields `name`,
   `version`, `when`, and `then`.
3. THE Prompt_Composer SHALL preserve each hook's `when` block unchanged, including `when.type` and
   any `toolTypes` filter.
4. THE Hook_Architecture_Improvements SHALL retain all `preToolUse` hooks with `toolTypes: ["write"]`
   (`gate-module3-visualization`, `enforce-mandatory-gate`, `write-policy-gate`).
5. WHEN deduplication is complete, THE `sync_hook_registry.py --verify` step SHALL exit with code 0.
6. THE Hook_Architecture_Documentation SHALL record the design decision of whether fragment
   composition is folded into `sync_hook_registry.py` or implemented as a sibling script
   (e.g., `compose_hook_prompts.py`), and SHALL require the CI-verifiable round-trip regardless of
   which option is chosen.

## Theme C — Capture-Hook Install Reliability

### Requirement 9: [Theme C] Installer curated list matches the actual hook set

**User Story:** As a bootcamper, I want the hook installer to offer exactly the hooks that actually
exist, so that I do not get errors for removed hooks or silently miss current ones.

#### Acceptance Criteria

1. THE Hook_Installer SHALL NOT reference the consolidated hooks `enforce-file-path-policies`,
   `enforce-single-question`, or `block-direct-sql` in its curated hook list or essential set.
2. THE Hook_Installer SHALL include `write-policy-gate`, `session-log-events`, `module-recap-append`,
   `enforce-mandatory-gate`, and `enforce-gate-on-stop` in its curated hook list with accurate
   display names and descriptions.
3. THE Hook_Installer's curated hook set SHALL equal the set of `*.kiro.hook` filenames present in
   the power's hooks directory.
4. THE Hook_Installer's curated hook names SHALL each follow the "to {verb phrase}" pattern matching
   the `name` field of the corresponding `.kiro.hook` file.
5. WHERE feasible, THE Hook_Installer SHALL derive its hook set from `hook-categories.yaml` or the
   `.kiro.hook` files rather than from a separately hand-maintained duplicate list; the design
   document SHALL record whether this derivation is adopted.

### Requirement 10: [Theme C] Capture-critical hooks are covered on both install paths

**User Story:** As a bootcamper, I want the capture-critical hooks installed or my missing them
flagged, so that my completion summary and recap are not silently incomplete.

#### Acceptance Criteria

1. THE Hook_Architecture_Documentation SHALL designate `session-log-events`, `module-recap-append`,
   and `ask-bootcamper` as capture-critical hooks.
2. THE createHook-from-registry path SHALL create the capture-critical hooks as part of the hook set
   created during onboarding/session start.
3. THE Hook_Installer SHALL install all capture-critical hooks when the bootcamper selects the
   install-all option or the essential option.
4. IF any capture-critical hook is absent from the bootcamper's `.kiro/hooks` directory at session
   start, THEN THE Session_Start_Check SHALL warn the bootcamper which capture-critical hooks are
   missing and how to install them.
5. THE Hook_Architecture_Documentation SHALL state that capture-critical coverage is required on both
   the createHook-from-registry path and the `install_hooks.py` file-copy path.
6. WHERE auto-installation of capture-critical hooks is feasible given the IDE's `createHook`-based
   installation model, THE design document SHALL record whether auto-installation is adopted; the
   warn-on-absence behavior in criterion 4 SHALL be provided regardless.

### Requirement 11: [Theme C] Non-interactive installer mode

**User Story:** As a CI/automation author, I want to install hooks without interactive prompts, so
that capture hooks can be installed in automated environments.

#### Acceptance Criteria

1. THE Hook_Installer SHALL accept a non-interactive flag that installs all hooks without reading
   from standard input.
2. THE Hook_Installer SHALL accept a non-interactive flag that installs only the essential/critical
   hook set without reading from standard input.
3. WHEN a non-interactive flag is supplied, THE Hook_Installer SHALL NOT call `input()`.
4. WHEN a non-interactive installation completes successfully, THE Hook_Installer SHALL exit with
   code 0.
5. IF the power hooks directory is missing when a non-interactive flag is supplied, THEN THE
   Hook_Installer SHALL exit with code 1 and report the missing directory.
6. THE Hook_Installer SHALL retain its existing interactive mode when no non-interactive flag is
   supplied.

### Requirement 12: [Theme C] Tests that prevent curated-list staleness from recurring

**User Story:** As a power maintainer, I want automated tests that fail when the installer's known
hook set diverges from the actual hook files, so that the staleness problem cannot return unnoticed.

#### Acceptance Criteria

1. THE Installer_Test_Suite SHALL assert that the Hook_Installer's curated hook filename set equals
   the set of `*.kiro.hook` files in the power hooks directory.
2. THE Installer_Test_Suite SHALL assert that the Hook_Installer references none of the consolidated
   hooks `enforce-file-path-policies`, `enforce-single-question`, or `block-direct-sql`.
3. THE Installer_Test_Suite SHALL assert that every capture-critical hook
   (`session-log-events`, `module-recap-append`, `ask-bootcamper`) is present in the install-all set
   and the essential set.
4. THE Installer_Test_Suite SHALL assert that the non-interactive install-all and essential modes
   complete without reading from standard input and exit with code 0.
5. THE Installer_Test_Suite SHALL follow the repository's test conventions: pytest with Hypothesis
   for property-based tests, class-based organization, `st_`-prefixed strategies,
   `@settings(max_examples=20)`, `from __future__ import annotations`, and modern type hints.
6. THE Installer_Test_Suite SHALL be placed according to the repository convention that tests
   validating real hook files live in the repo-root `tests/` directory while script-behavior tests
   live in `senzing-bootcamp/tests/`.
