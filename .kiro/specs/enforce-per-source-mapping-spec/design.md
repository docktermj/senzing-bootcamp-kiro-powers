# Design: Enforce Per-Source Mapping Specification

## Overview

This fix adds hook-based enforcement to guarantee that a per-source mapping specification markdown file (`docs/{source_name}_mapper.md`) is created for every data source that goes through the Module 5 mapping workflow. The existing steering instructions tell the agent to create these files, but the agent doesn't always follow through. Hooks provide a structural enforcement mechanism that fires automatically and blocks progression.

## Architecture

### Enforcement Strategy

Two hooks working together:

```
Hook 1: enforce-mapping-spec (fileCreated → askAgent)
  Trigger: data/transformed/*.jsonl, data/transformed/*.json
  Action: Check if docs/{source_name}_mapper.md exists
  If missing: Instruct agent to create it NOW before any other work
  If present: Produce no output (silent pass)

Hook 2: Enhanced analyze-after-mapping (existing hook, updated prompt)
  Trigger: data/transformed/*.jsonl, data/transformed/*.json
  Action: EXISTING quality validation + NEW mapping spec verification
  Combined check: quality AND mapping spec must both pass
```

### Why Two Mechanisms?

- **Hook 1 (enforce-mapping-spec)**: Dedicated enforcement hook with a clear, single responsibility. Fires on the same trigger as analyze-after-mapping but focuses exclusively on the mapping spec file.
- **Hook 2 (enhanced analyze-after-mapping)**: The existing hook already fires on the right trigger. Adding a mapping spec check to its prompt provides defense-in-depth — even if Hook 1 is somehow bypassed, the quality validation hook also reminds about the mapping spec.

## Components and Interfaces

### Component 1: enforce-mapping-spec Hook

```json
{
  "name": "Enforce Mapping Specification",
  "version": "1.0.0",
  "description": "When transformed data is created, verifies that a per-source mapping specification markdown exists in docs/. Blocks progression until the mapping spec is created.",
  "when": {
    "type": "fileCreated",
    "patterns": ["data/transformed/*.jsonl", "data/transformed/*.json"]
  },
  "then": {
    "type": "askAgent",
    "prompt": "..."
  }
}
```

**Prompt logic:**

```
A transformed data file was just created in data/transformed/.

REQUIRED CHECK: Extract the source name from the filename (e.g., "customers" from "customers.jsonl"). Check if the file docs/{source_name}_mapper.md exists.

If docs/{source_name}_mapper.md ALREADY EXISTS:
  Produce no output. STOP. Zero tokens. The mapping spec is in place.

If docs/{source_name}_mapper.md DOES NOT EXIST:
  You MUST create it NOW before doing any other work. Do not proceed to the next data source. Do not proceed to quality validation. Do not proceed to Module 6.

  Create docs/{source_name}_mapper.md with this structure:

  # Mapping Specification: {SOURCE_NAME}

  **Source file:** data/raw/{source_file}
  **Data source name:** {DATA_SOURCE}
  **Entity type:** Person / Organization / Both
  **Generated by:** mapping_workflow

  ## Field Mappings

  | Source Field | Senzing Attribute | Transformation | Notes |
  |---|---|---|---|
  | (fill from the mapping workflow results) |

  ## Mapping Decisions

  - (key decisions made during mapping)

  ## Quality Notes

  - (quality observations specific to this source)

  Fill in the actual field mappings from the mapping workflow you just completed. This file must be self-contained — a developer reading only this file should be able to recreate the transformation program in any language.
```

### Component 2: Enhanced analyze-after-mapping Prompt

Update the existing `analyze-after-mapping.kiro.hook` prompt to add a mapping spec reminder at the end:

```
[existing quality validation prompt]

ADDITIONALLY: After quality validation, verify that docs/{source_name}_mapper.md exists (extract source name from the transformed filename). If it does not exist, remind the agent: "The per-source mapping specification is missing. Create docs/{source_name}_mapper.md before proceeding."
```

### Component 3: Module 5 Completion Gate (Steering Enhancement)

In `module-05-phase2-data-mapping.md` Step 12, strengthen the completion checkpoint:

```
Before marking Module 5 as complete, list ALL files in data/transformed/ and verify that EACH has a corresponding docs/{source_name}_mapper.md. If any are missing, create them now. Do not write the module completion checkpoint until all mapping specs exist.
```

This is a steering instruction (not a hook), but combined with the hooks it provides triple enforcement.

## Data Models

No data model changes. The mapping spec files follow the existing template already defined in `module-05-phase2-data-mapping.md` Step 11.

### File Naming Convention

| Transformed File | Expected Mapping Spec |
|---|---|
| `data/transformed/customers.jsonl` | `docs/customers_mapper.md` |
| `data/transformed/vendors.jsonl` | `docs/vendors_mapper.md` |
| `data/transformed/toybox_stores.jsonl` | `docs/toybox_stores_mapper.md` |

The source name is extracted by stripping the directory prefix and file extension from the transformed filename.

## Error Handling

- If the transformed filename contains characters that are invalid for a markdown filename, sanitize them (replace spaces with underscores, remove special characters)
- If the agent cannot determine the field mappings (e.g., the mapping workflow state was lost), it should create the mapping spec with placeholder content and flag it for the bootcamper to review
- If the hook fires but the agent is not in Module 5 (e.g., re-running a transformation in a later module), the hook should still enforce the mapping spec — the requirement is unconditional

## Testing Strategy

### Structural Verification

1. Verify `enforce-mapping-spec.kiro.hook` exists with correct trigger patterns
2. Verify the hook prompt contains the mapping spec template
3. Verify the hook prompt contains the "produce no output if file exists" silent-pass rule
4. Verify `analyze-after-mapping.kiro.hook` prompt includes the mapping spec reminder
5. Verify `hook-registry.md` contains the new hook entry

### Behavioral Verification

1. When a transformed file is created and the mapping spec exists → hook produces no output (silent pass)
2. When a transformed file is created and the mapping spec does NOT exist → hook instructs agent to create it
3. The mapping spec created by the agent contains: source file, data source name, entity type, field mappings table

### Regression Verification

1. The `analyze-after-mapping` hook still performs quality validation via `analyze_record`
2. Hooks do not fire for files outside `data/transformed/`
3. Hooks do not interfere with modules other than Module 5
